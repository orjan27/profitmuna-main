# Project Structure

- Never create new top-level directories. Allowed roots are listed in `CLAUDE.md` under "Project Structure (STRICT)".
- All `.md` documentation files go in `docs/`. Only `CLAUDE.md`, `README.md`, `AGENTS.md`, and `STANDARDS.md` are allowed at the project root.
- New files must live in one of the folders listed in `CLAUDE.md`. If a use case doesn't fit, update `CLAUDE.md` first — don't improvise a folder.
- Do not create sibling folders like `utils/`, `helpers/`, `common/`, or `shared/` inside `apps/*/src/`. They duplicate the existing `lib/` and `types/` folders.
- Mirror the established per-app layout exactly — Hono apps use `routes/`, `middleware/`, `services/`, `schemas/`, `lib/`, `types/`; Spring Boot apps use `controller/`, `service/`, `repository/`, `model/`, `config/`.
- Place shared TypeScript types in `types/`. Place shared Java model classes in `model/`. Do not scatter type definitions across implementation files.
- Do not add new files under `.claude/hooks/`, `.claude/skills/`, `.claude/rules/`, or `.claude/commands/` without the user's explicit request.
- Route handlers stay thin. Put business logic in `services/` (Hono) or `service/` (Spring). HTTP concerns (`c.req`, `c.json`, `HttpServletRequest`) never appear inside services.
- Prefer path aliases (`@/*`, `@app/db`) over relative imports that climb more than one level.
- `components/ui/*` is generated by shadcn. Do not hand-edit — regenerate via `/generate-component`.
