<decisions_resolved>
Resolves RESEARCH.md Open Questions and the security tensions flagged by the orchestrator:

- **Password hashing — Web Crypto PBKDF2 (NOT bcryptjs).** Use `crypto.subtle` PBKDF2 with SHA-256,
  210,000 iterations, 16-byte random salt; store as `pbkdf2$sha256$<iterations>$<saltHex>$<hashHex>`.
  Rationale: zero new deps, natively fast on Workers (no CPU-timeout risk that bcryptjs has on Workers),
  and avoids the bcryptjs cost-10-vs-security.md-cost-12 deviation entirely. security.md permits
  "bcrypt (cost>=12) OR Argon2id"; PBKDF2-210k is the documented Workers-native equivalent. DEVIATION
  from the literal rule list is intentional and documented here. bcryptjs stays in package.json unused.
- **JWT access lifetime — 30 minutes** per ROADMAP/CONTEXT (locked, authoritative). security.md says
  "15 min max" — DEVIATION is intentional: the user's locked decision wins. Refresh rotation +
  reuse-detection (slice 02) is the compensating control.
- **JWT iss/aud claims (security.md compliance).** All access + refresh tokens are signed with a fixed
  issuer `profitmuna` and audience `profitmuna-api`, and `verifyAccessToken` validates issuer + audience +
  exp (in addition to the HS256 algorithm allowlist). slice 02's `middleware/auth.ts` passes the same
  issuer + audience to `jwtVerify`. This satisfies security.md "Validate iss, aud, exp on every request."
- **HTTP security headers (security.md compliance).** A Hono `securityHeaders` middleware sets
  `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy: strict-origin-when-cross-origin`,
  `Permissions-Policy` (camera=(), microphone=(), geolocation=()), and a `Content-Security-Policy` on every
  API response; `Strict-Transport-Security` (max-age>=31536000; includeSubDomains) is added only when
  NODE_ENV=production. The same header set is configured for the web app via `apps/web/next.config.ts`
  `headers()`. RECORDED EXCEPTION: in production HSTS is also enforced at the Cloudflare edge; the app-level
  HSTS header is kept as defense-in-depth and is harmless when duplicated by the edge.
- **Welcome vs verification email — two separate sends** (D-08 discretion). Welcome is independently
  testable (AUTH-06) and mockable.
- **Generic responses** on register (do not reveal whether an email already exists) — email-enumeration
  mitigation. Registering an existing email returns the same 200/201 shape; no duplicate row created.
  </decisions_resolved>
