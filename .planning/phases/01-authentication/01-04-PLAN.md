<decisions_resolved>

- **Wave placement (wave 4, runs after 01-03):** 01-04 depends only on 01-02 (like 01-03), so by
  dependency alone it could be wave 3. It is deliberately placed at wave 4 because it shares three files
  with 01-03 — apps/api/src/services/auth-service.ts, apps/api/src/routes/auth.ts, apps/api/tests/auth.test.ts.
  GSD requires same-wave plans to have zero files_modified overlap; running 01-03 and 01-04 in the same
  wave would cause concurrent edits to the shared auth service/routes/tests. Serializing 01-04 after
  01-03 keeps file ownership clean. This is the documented, intentional reason the wave is 4 and not 3.
- **Account linking by email (Claude's discretion default):** Google emails are pre-verified, so on
  callback: if a user with that email exists, set its googleId and emailVerified=true (link); otherwise
  create a new user with googleId, emailVerified=true, passwordHash=null. A returning Google user matches
  by googleId. No duplicate accounts; no error on collision.
- **Session issuance:** the callback issues the same access (30m, iss/aud) + refresh (7d, DB-stored rotating)
  httpOnly cookies as password login, then redirects to APP_BASE_URL '/' (dashboard).
- **State/PKCE cookies:** SameSite=Lax (must survive the cross-site OAuth redirect — Pitfall 6), httpOnly,
  10-min maxAge, Secure in prod. Validate state equality + presence of code_verifier before token exchange.
- **dep install:** `arctic@3.7.0` was installed in slice 01 (Task 0 legitimacy gate already covered it).
  </decisions_resolved>

## Trust Boundaries

| Boundary                          | Description                                                    |
| --------------------------------- | -------------------------------------------------------------- |
| browser -> Google -> API callback | OAuth redirect chain; state + PKCE verifier cross the boundary |
| API -> Google token/userinfo      | Server-to-server token exchange + userinfo fetch               |
| API -> D1                         | upsert user; insert refresh token                              |

## STRIDE Threat Register

| Threat ID | Category               | Component                          | Disposition | Mitigation Plan                                                                                                                            |
| --------- | ---------------------- | ---------------------------------- | ----------- | ------------------------------------------------------------------------------------------------------------------------------------------ |
| T-04-01   | Spoofing / Tampering   | OAuth callback CSRF                | mitigate    | arctic state generated + stored as httpOnly Lax cookie; callback rejects state mismatch (400 invalid_oauth_state)                          |
| T-04-02   | Spoofing               | open redirect on callback          | mitigate    | Redirect only to fixed APP_BASE_URL '/'; never to a user-supplied URL (security.md)                                                        |
| T-04-03   | Elevation of privilege | account-linking takeover           | mitigate    | Link only on exact email match; Google emails are provider-verified; set emailVerified=true only via Google identity, never via user input |
| T-04-04   | Tampering              | PKCE downgrade / code interception | mitigate    | arctic generateCodeVerifier + validateAuthorizationCode (PKCE); verifier stored httpOnly, consumed once                                    |
| T-04-05   | Information disclosure | leaking Google tokens              | accept      | Google access token used transiently for userinfo, never stored; acceptable for v1                                                         |
| T-04-SC   | Tampering              | arctic package                     | mitigate    | Verified in slice 01 Task 0 legitimacy gate; no new install here                                                                           |
