<decisions_resolved>

- **Refresh token storage:** the refresh COOKIE carries an opaque random token (generateSecureToken),
  whose sha256 hash is the refresh_tokens row. (We do NOT use the jose 7d JWT as the refresh credential —
  the opaque-token-in-DB model is what enables rotation + reuse-detection per D-11. signRefreshToken
  remains available but the DB-backed opaque token is the source of truth for refresh.)
- **Access token:** jose HS256 JWT, 30m, sub=userId, iss 'profitmuna', aud 'profitmuna-api', set as
  httpOnly cookie `access_token`.
- **JWT iss/aud validation in middleware (security.md):** requireAuth verifies the access token via
  verifyAccessToken (which passes `issuer` + `audience` + `algorithms:['HS256']` to jwtVerify). The
  middleware never re-implements verification with a looser option set — it must reuse the slice-01
  verifyAccessToken so iss/aud/exp are always validated. A token missing/mismatching iss or aud -> 401.
- **Cookies:** httpOnly, SameSite=Lax, Secure when NODE_ENV=production, path '/' (D-02). access maxAge 1800s, refresh maxAge 604800s.
- **Rate limiting:** D1 login_attempts counter per email (Open Question 3). On >=5 failures within the
  window, lock for 15m and return 429 with Retry-After (security.md). Non-atomic race accepted for v1;
  documented. Reset counter on successful login.
- **Logout = global revocation:** delete all refresh_tokens rows for the user (D-12 "log out everywhere").
  </decisions_resolved>

<threat_model>

## Trust Boundaries

| Boundary            | Description                                                |
| ------------------- | ---------------------------------------------------------- |
| browser -> BFF      | Login credentials + cookie-bearing authenticated calls     |
| BFF -> API /refresh | Silent server-to-server refresh; relays rotated cookies    |
| API -> D1           | refresh_tokens rotation/revocation; login_attempts counter |

## STRIDE Threat Register

| Threat ID | Category               | Component                         | Disposition | Mitigation Plan                                                                                                                                     |
| --------- | ---------------------- | --------------------------------- | ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------- |
| T-02-01   | Elevation of privilege | refresh token theft / reuse       | mitigate    | DB-stored sha256 hashes; rotation on every refresh; reuse of a revoked token revokes the whole chain for the user (D-11)                            |
| T-02-02   | Tampering              | brute-force login                 | mitigate    | login_attempts D1 counter, 5-fail/15-min lockout, 429 + Retry-After (security.md). Non-atomic race documented                                       |
| T-02-03   | Spoofing               | JWT algorithm/claim confusion     | mitigate    | verifyAccessToken uses algorithms:['HS256'] allowlist + validates issuer + audience + exp; requireAuth reuses it per-request from c.env (Pitfall 4) |
| T-02-04   | Information disclosure | XSS cookie theft                  | mitigate    | access + refresh cookies are httpOnly; never read via document.cookie; SameSite=Lax                                                                 |
| T-02-05   | Tampering              | CSRF on state-changing auth POSTs | mitigate    | SameSite=Lax + same-origin BFF: cross-site form posts cannot send the cookie / hit the API directly                                                 |
| T-02-06   | Information disclosure | failed-login response             | mitigate    | Generic 401 invalid_credentials (do not distinguish "no such user" vs "wrong password")                                                             |
| T-02-07   | Spoofing               | logout incompleteness             | mitigate    | logout deletes all refresh_tokens rows for the user (server-side revocation) AND clears both cookies (D-12)                                         |

</threat_model>
