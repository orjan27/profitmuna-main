<objective>

This delivers WAL-04 (manual transactions + edit + soft-delete + restore) and WAL-05 (detail view + paginated history). After this plan, every Phase 4 success criterion is satisfied: a user can fully manage a wallet end-to-end.

## Trust Boundaries

| Boundary                          | Description                                                                                                         |
| --------------------------------- | ------------------------------------------------------------------------------------------------------------------- |
| browser → BFF proxy → Workers API | transaction create/edit/delete/restore payloads + page query cross from untrusted client to the authorization point |
| Workers API → D1                  | transaction mutations + cross-table history reads; userId scoping + double-count guard enforced here                |

## STRIDE Threat Register

| Threat ID | Category               | Component                                                                            | Disposition | Mitigation Plan                                                                                                                                        |
| --------- | ---------------------- | ------------------------------------------------------------------------------------ | ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| T-04-11   | Elevation of Privilege | transaction CRUD / detail on another user's wallet (IDOR)                            | mitigate    | getById + every transaction method scoped with `eq(wallets.userId, userId)` / `eq(walletTransactions.userId, userId)` from the verified JWT (Task 1)   |
| T-04-12   | Tampering              | balance double-count via manual deposit + auto income (or withdrawal + auto expense) | mitigate    | `assertCanInsertTransaction` runs server-side before insert; cannot be bypassed by the client (Task 1)                                                 |
| T-04-13   | Tampering              | negative or zero transaction amount                                                  | mitigate    | `amount: z.number().positive()` in walletTransactionSchema (Plan 01); service rejects toCents(amount) ≤ 0 (Task 1)                                     |
