<objective>
Close the single SC-5 gap from 04-VERIFICATION.md: paginated transaction history in `getById()` is non-deterministic and under-counted once any source exceeds one page. The three per-source history queries have no `ORDER BY`, so `.limit(fetchLimit)` captures an arbitrary subset under D1; `total = merged.length` understates `totalPages`; and the `autoDeductAllExpenses` path fans out one query per category (N+1) with the same ordering flaw. This plan also co-fixes two review findings that touch the same files: CR-02 (unvalidated numeric path params → wrong HTTP code) and WR-04 (NewWalletForm fires a success toast in the catch block for ALL throws, masking real errors as success).

Purpose: WAL-05 becomes correct under real data volumes (> 20 rows/source), and route-level input validation + form error handling match project conventions (api-routes.md, error handling via toast).

Output: Ordered + COUNT-backed paginated history, a single inArray expense-history query, Zod param validation returning 422, a redirect-aware NewWalletForm catch block, and regression tests that fail on the current code and pass after the fix.
</objective>
