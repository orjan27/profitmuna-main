<objective>
Close VERIFICATION gap 2 (CR-01): `updatePercentages` validates only the submitted subset, so an authenticated client can PUT a single account with `targetPercentage: 10000`, pass the sum check, and leave the persisted allocation set summing to more than 10000 bp — silently over-allocating income and breaking the core product invariant.

Purpose: Move the sum-to-100% invariant from a client-trust contract to a server-enforced one. The comment at line 382-385 ("The caller is responsible for submitting ALL accounts") is exactly the anti-pattern security.md warns against.
Output: A hardened `updatePercentages` that fetches user-owned account IDs, requires the submitted set to cover exactly those IDs (no missing, no foreign, no duplicates), then validates `sum === 10000`. Plus a regression test asserting a partial submission returns 400.
</objective>
