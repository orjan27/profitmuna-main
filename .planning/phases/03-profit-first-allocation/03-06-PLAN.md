<objective>
Close VERIFICATION gap 1 (WR-01): the income-category filter is backend-wired but frontend-unreachable. `page.tsx` renders `<PfFilters />` with no `categoryOptions`, so the prop defaults to `[]`, the category Sheet is gated on `categoryOptions.length > 0`, and the user has no entry point to set `categoryIds` — even though the API and URL plumbing fully implement category filtering.

Decision (per gap-closure guidance): A real category data source exists in the codebase — the `income_categories` table and the `incomes` table's denormalized `categoryName` column (verified in packages/db/src/schema.ts). The fix derives real category options from data the user already has, served through the API (web never touches the DB directly). `getSummary` is extended to also return the DISTINCT `(categoryId, categoryName)` pairs present in the user's RECEIVED + profitFirstAllocated income; `page.tsx` maps those into `categoryOptions` and passes them to `<PfFilters categoryOptions={...} />`. When the user has no qualifying income categories yet, an explicit empty-state replaces the dead-prop silence.

Purpose: Make SC4's "filterable by income category" reachable end-to-end with real labels, without depending on an as-yet-unplanned Phase 2 categories endpoint.
Output: A category multi-select that appears with real category labels and filters the summary; an explicit empty-state when no categories are present.
</objective>
